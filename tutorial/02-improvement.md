# 02 提分教程：让 Agent 迭代得更有效

评分是赛后统一进行，没有实时 leaderboard。你唯一能依赖的是本地指标。每次迭代后记录指标变化，这是唯一可靠的验证方式。

---

## 一、实验记录

每次迭代固定记录实验 ID、假设 ID、改动内容、改动文件、最佳结合能、平均结合能、**trivial route** 比例、路线平均步数、Agent 决策。

Agent 自动写入 **docs/iteration_log.jsonl**，你也可以手动维护一个表格对照：

```
实验ID | 假设ID | 改动内容         | 最佳结合能 | 平均结合能 | trivial比例 | 结果
E001   | -      | baseline         | -7.2       | -6.1       | 2/10        | 基线
E002   | H001   | 扩充骨架库+10个  | -7.5       | -6.4       | 2/10        | 有效
E003   | H002   | 修正对接盒子坐标 | -8.1       | -7.2       | 1/10        | 有效
```

---

## 二、优化 Agent 本身（优先）

化学指标的提升应该由 Agent 自主发现和实现。首要任务是让 Agent 的迭代循环更高效。

### 方向 1：优化 program.md 的指令质量

program.md 是 Agent 的唯一指令书。指令越清晰，Agent 的决策越高效。常见优化点有四个：

一是细化假设模板。在阶段二要求 Agent 明确写出验证指标和失败标准，避免假设模糊不清。

二是补充领域知识。在 program.md 中补充药物化学常识，例如含氮杂环通常有更好的结合能力，磺酰胺是常见的氢键供体。这些知识会直接影响 Agent 提出的假设质量。

三是调整迭代策略。如果 Agent 总是提过于宏大的假设，在 program.md 中增加低风险原则的强调；如果验证失败后不分析原因就换方向，增加失败分析的强制要求。

四是指导工具使用。告诉 Agent 在什么情况下应该用 SearchWeb 搜索最新方法，什么情况下应该重新阅读 papers/ 中之前忽略的章节。

在 program.md 的阶段二追加以下内容，可以显著提升假设质量：

```markdown
提出假设时，必须同时回答：
1. 如果核心指标提升小于 5%，该假设是否仍值得保留？
2. 如果指标下降，最可能的原因是什么？（代码 bug / 假设不成立 / 验证方式问题）
3. 该假设的最低可接受结果是什么？
```

### 方向 2：让 Agent 善用 SearchWeb

Agent 诊断陷入困境时，在 program.md 中增加：

```markdown
连续两个假设验证失败，必须执行以下之一：
- 使用 SearchWeb 搜索该瓶颈的最新解决方案
- 重新阅读 papers/ 中之前忽略的章节
- 换一个完全不同的瓶颈方向
```

### 方向 3：优化实验设计

增加对照实验要求：

```markdown
验证假设时，必须同时运行：
1. 修改后的版本（实验组）
2. 未修改的版本（对照组）
3. 对比指标：最佳结合能、top10 平均结合能、QED 均值、trivial route 比例
```

## 三、优化底层引擎（Agent 改不动时你再出手）

连续 3 轮没有有效进展，可以手动优化底层引擎，给 Agent 更好的起点。

容易上手的方向有三个。一是验证并修正对接盒子坐标。config.py 里的 DOCKING_CENTER = [18.28, 2.31, 21.44] 是预设值，如果偏离真实活性位点，所有对接打分都不可信。用 PyMOL 或 ChimeraX 打开 target.pdb，找到活性口袋，计算几何中心后更新 config.py，再让 Agent 重新跑一轮对比结合能变化。二是扩充骨架库。generator.py 的 SCAFFOLDS 列表只有 27 个，覆盖面有限。每次添加 5 到 10 个新骨架，让 Agent 验证效果。三是调整过滤阈值。MIN_QED = 0.3 比较宽松，可以尝试 0.4 或 0.5，MAX_MW 调到 450。每次只改一个参数。

有点门槛的方向有两个。一是扩充逆合成规则库。synthesis_v2.py 有 35+ 条规则，但 trivial route 仍然会出现。继续扩充规则，尤其是稠环、桥环的断键规则。二是引入基于口袋的定向生成。分析 target.pdb 的口袋特征（疏水区、氢键供体/受体位置），在 generator.py 中增加口袋感知策略。

进阶方向有两个。一是升级分子生成模型，用 DiffSBDD、TargetDiff 等扩散模型替换 RDKit 变异生成器，需要 GPU。二是引入主动学习迭代，第一轮跑完后分析高分分子的共同特征，以这些特征为种子进行第二轮定向生成。

---

## 四、核心原则

Agent 优先。先让 Agent 自己迭代，你只在它遇到瓶颈时出手。

一次只改一个变量。无论是改 program.md 还是底层代码，一次只改一个地方。

记录前后对比。没有对比的改动等于没做。

过程即分数。即使某轮假设被证伪，完整的诊断到假设到验证到分析过程本身就是 Agent 能力的证明。

不要伪造结果。实验失败要如实记录，失败同样是科学发现的一部分。

---

## 五、常见问题

Agent 运行到一半卡住了怎么办？

检查 docs/iteration_log.jsonl 最后一条记录，看卡在哪一步。常见原因有三种：代码修改引入语法错误，Agent 会在 code_evolution 阶段卡住，需要人工修复 src/ 中的代码；对接运行时间过长，可以减小 --n-generate 参数；LLM API 响应慢，这是正常现象，耐心等待。

Agent 总是提出同一个假设怎么办？

在 program.md 中增加约束：已被 REJECTED 的假设，除非有新的文献支撑，否则不得重复提出。

Agent 改的代码把项目搞崩了怎么办？

Agent 在修改前会自动 git commit 备份。你可以执行 git log --oneline -5 查看历史，然后 git reset --hard HEAD~1 回退到上一次提交。

一轮迭代要多久？比赛时间够吗？

一轮约 30 到 60 分钟。关键是每轮都要有清晰的假设和验证，不是盲目多跑。

Agent 能力的分数占比很高。哪怕化学指标一般，只要 Agent 展现出真实的自主科研过程，总分的竞争力就会很强。
